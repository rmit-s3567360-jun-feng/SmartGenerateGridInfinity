# Open Requirements

更新时间：2026-03-22

这份文档用于沉淀“已讨论、已确认方向、但尚未完全收口”的需求，供后续开发、交接和 AI 上下文传递使用。

状态标记：

- `已实现`：当前代码已经落地，可在仓库中直接找到对应实现
- `部分实现`：已经有一部分能力，但仍未达到当前期望
- `开放需求`：已确认方向，但尚未完成开发
- `暂缓`：已讨论，但当前不进入近期排期

后续新增的未完成需求，默认继续追加到本文件，而不是分散到新的需求文档。

## 1. 当前代码基线

以下内容为当前真实已落地能力，用于和开放需求区分：

- 模板数量：6 个
  - `generic-bin`
  - `parametric-cavity-bin`
  - `memory-card-tray`
  - `photo-outline-bin`
  - `stl-cavity-bin`
  - `stl-retrofit`
- 生成器主路由：`/generator/:templateId`
- 参数系统：已经不再局限于标量参数
  - 仍支持 `string | number | boolean`
  - 也已支持数组与结构化对象参数
  - 当前已落地的复杂参数例子包括：
    - `shapeEntries`
    - 照片轮廓 `analysis`
    - 导入 STL 的 `source` 摘要
- 几何生成：浏览器内 Worker 执行
- 图像识别：照片轮廓模板当前仍在主线程执行，尚未拆独立识别 Worker
- 导出格式：仅支持 `STL`
- 3D 预览：`three.js`
- 预览坐标轴：已支持 `X / Y / Z` 坐标轴
- 预览原点：当前位于模型外轮廓角点，不再与模型中心点重合

当前模板和通用能力状态：

- `通用收纳盒`
  - `已实现`
  - 支持 `X / Y / Z` 三个方向内壁厚度调整
  - 支持按“从最外侧内壁到隔板起始边”的语义定义隔板距离
  - 支持统一设置隔板厚度和隔板高度
  - 当前已回归为纯隔板模板
- `参数化型腔盒`
  - `已实现`
  - 模板 id 为 `parametric-cavity-bin`
  - 支持 `rectangle / rounded-rectangle / circle / capsule`
  - 支持自动排布、固定尺寸校验、固定 `X / Y / Z` 手动旋转
  - 支持 `x-first / y-first` 两种排列方式
  - 支持手动点击“生成图形”以及单件复制 / 粘贴 / 粘贴新增
  - 型腔默认居中，并保持多个独立开口
- `内存卡托盘`
  - `已实现`
  - 当前稳定模式仅保留：
    - `microSD 极限收纳`
    - `SD 紧凑收纳`
    - `混合收纳`
  - 之前讨论过的竖放模式已取消，不视为当前需求基线
- `照片轮廓收纳`
  - `部分实现`
  - 已支持上传照片、L 形标尺校准、关键点轮廓编辑、尺寸审查和自动外部尺寸搜索
  - 仍缺“取物凹槽”能力
- `STL 型腔收纳`
  - `已实现`
  - 已支持导入被收纳物 STL、90° 步进旋转、自动推荐外部尺寸、真实 STL 负形布尔
- `STL 改底适配`
  - `已实现`
  - 语义已与“STL 型腔收纳”分离，保留为标准矩形实体规整工作流

历史模板状态：

- `screwdriver-rack`
  - 不再作为当前产品入口保留
- `pliers-holder`
  - 不再作为当前产品入口保留
  - 对应升级需求继续以“钳类模板 V2”维护

开发和发布辅助能力：

- `已实现`：Windows 一键启动脚本 `scripts/start-dev.bat`
- `已实现`：IDE 可调用命令 `npm run dev:open`
- `已实现`：Linux 云服务器构建脚本 `scripts/build-release.sh`

## 2. 已确认需求的当前状态

以下条目按当前仓库实际状态维护；它们并不都是“完全未做”，其中包含 `部分实现` 与 `开放需求`。

### 2.1 照片轮廓收纳模板

状态：`部分实现`

当前已落地：

- 支持拍照上传
- 基于照片提取待收纳物品的外轮廓
- 自动生成最合适大小的 Gridfinity 盒子
- 提供尺度校准与尺寸审查信息
- 支持关键点拖拽编辑

当前未完成：

- 可选在 `X` 轴两侧添加便于取物的双层凹槽
- 独立图像识别 Worker

备注：

- 这是当前仍需要完整保留细节的部分实现需求
- 详细规格见第 3 节

### 2.2 STL 型腔收纳与旧收纳盒迁移

状态：`部分实现`

当前已落地：

- 支持把“被收纳物 STL”直接转成新的 Gridfinity 型腔收纳盒
- 最终结果是标准矩形 Gridfinity 盒体，而不是纯实心规整块
- 该工作流已经独立为 `stl-cavity-bin`

当前未完成：

- 支持把“旧收纳盒 STL”中的已有负形提取出来，再迁移成新的 Gridfinity 型腔收纳盒
- 为“旧收纳盒迁移”新增独立模板 `legacy-organizer-migrate`

当前边界：

- “被收纳物 STL -> 型腔收纳盒”已完成首版
- “旧收纳盒迁移”仍作为第二阶段，基于型腔提取能力继续扩展
- 当前 `stl-retrofit` V1 只是“标准矩形实体规整”，不属于这条需求的完成态

详细规格：

- 见 `docs/STL_CAVITY_WORKFLOWS_SPEC.md`

### 2.3 钳类模板 V2

状态：`开放需求`

目标：

- 把历史上的“钳子收纳”升级为更通用的“钳类 / 手持夹持工具收纳”
- 用户只需要调整少数核心参数，就能适配更多钳类和类似工具
- 模板应从“固定矩形槽”升级为更通用的限位型或渐缩型槽道

建议的核心参数方向：

- 数量
- 最大宽度
- 最大厚度
- 插入长度
- 前部开口宽度

### 2.4 更复杂的非标输入能力

状态：`部分实现`

当前已落地：

- 上传文件后参与几何求解的复杂参数结构
- 在模板里传递轮廓点数组或图像分析结果
- 轮廓拖点编辑型的专用模板界面

当前未完成：

- 独立的图像识别 Worker
- 更通用、可复用的“文件输入 -> 分析结果 -> 几何求解”公共基础设施

说明：

- 这些能力已经不再完全属于“未来想法”
- 当前主要是还没有进一步抽象成通用层，且图像识别仍未搬进独立 Worker

### 2.5 参数化型腔盒模板 V1

状态：`已实现`

当前已落地：

- 新增独立模板 `parametric-cavity-bin`
- `generic-bin` 回归纯隔板模板，不再承载型腔模式切换
- 支持基础参数化形状：
  - `rectangle`
  - `rounded-rectangle`
  - `circle`
  - `capsule`
- 支持每种形状单独设置尺寸、数量，并自动生成可收纳这些形状的 Gridfinity 盒体
- 自动搜索最小可用外部尺寸，并允许用户通过固定 `X / Y / Z` 旋转显式切换姿态
- 支持 `x-first / y-first` 两种排列方式
- 支持手动点击“生成图形”，避免调参时每次都自动重算
- 支持单件复制 / 粘贴 / 粘贴新增，简化重复录入

最近一轮收口：

- 已移除旧的自动旋转与姿态限制 UI，当前只保留固定 `X / Y / Z` 手动旋转
- 摘要区与求解结果改为展示排列规则，不再表达“是否自动立放”
- 详细边界、错误文案和回归场景统一收口到 `docs/PARAMETRIC_SHAPE_CAVITY_SPEC.md`

首版边界：

- 同一条形状 entry 的全部数量共用一个最终姿态
- 若需要同一种形状混用多种姿态，用户通过新增第二条 entry 表达
- 型腔默认都是独立孔位，不合并成长槽

首版不做：

- 任意多边形、SVG、照片轮廓或 STL 作为输入
- 手工拖拽摆位、手工指定孔位坐标
- 接近 CAD 编辑器的自由编辑工作流

详细规格：

- 见 `docs/PARAMETRIC_SHAPE_CAVITY_SPEC.md`

### 2.6 参数列表界面 V2 与超宽屏适配

状态：`已实现`

当前已落地：

- 生成器左侧参数区已从扁平字段列表升级为分段式参数面板
- 参考 `Extrabold Gridfinity Baseplate` 的信息层级，补齐了 section、卡片化选择、整行开关和底部主操作
- 已完成 `2K / 超宽屏` 布局适配，减少大屏下的无效留白并提升预览区利用率

实现范围：

- 通用参数面板按 section 渲染，支持 `基础尺寸 / 内部结构 / 附加功能 / 高级设置` 等分组
- 已支持卡片式单选、segmented control、整行 `switch-row`、`X / Y / Z` 成组输入和悬浮提示
- 桌面端参数区保留 sticky 左栏与底部主操作，兼容自动生成模板和参数化型腔盒的双态流程
- `generic-bin`、`memory-card-tray`、`parametric-cavity-bin` 已完成首轮新参数面板接入

超宽屏适配结果：

- 已补齐 `>= 1600px` 和 `>= 1920px` 两档布局策略
- 生成器页最大宽度已放宽，超宽屏下控制栏、预览区和摘要区占比重新平衡
- 左侧控制栏在大屏下允许更宽，右侧内容区同步拉开，但不新增第三列业务区
- 帮助文案和说明信息继续保持阅读宽度约束，避免出现过长文本行

边界与非目标：

- 当前轮次不新增 `Share`、`3MF`、打印机预设等参考站能力
- 不改变现有 `/generator/:templateId` 路由与 `TemplateDefinition + zod + Worker` 架构
- 照片轮廓、`STL` 导入等专用工作流继续保留独立界面，不强行回收进通用表单
- 不新增新的几何参数，只优化现有参数的表达方式、信息层级和布局适配

验收标准：

- `generic-bin`、`memory-card-tray`、`parametric-cavity-bin` 至少完成首轮新参数面板接入
- 参数区不再是连续字段堆叠，用户可按 section 快速扫描并定位常用项
- 至少一类枚举参数使用卡片式或 segmented 呈现，布尔参数改为整行开关表达
- `visibleWhen`、高级折叠、`onReset`、参数校验和手动生成流程不回归
- 在 `1440px`、`1600px`、`1920px`、`2560px` 下都不出现明显的大面积空白、预览区过窄或超长文本行
- `1120px` 以下现有单列/移动端布局不回归

## 3. 照片轮廓收纳模板 V1 详细规格

状态：`部分实现`

当前实现概览：

- 已实现：独立模板、上传与轮廓审查界面、L 形标尺识别、关键点拖拽修正、`0° / 90°` 自动尺寸搜索
- 未实现：取物凹槽、独立图像识别 Worker

### 3.1 模板定位

- 模板 id：`photo-outline-bin`
- 这是一个独立模板，与当前 6 个模板中的其余模板并列
- 当前并列模板包括：
  - `generic-bin`
  - `parametric-cavity-bin`
  - `memory-card-tray`
  - `photo-outline-bin`
  - `stl-cavity-bin`
  - `stl-retrofit`

### 3.2 输入与识别边界

首版固定为：

- 纯前端识别
- 用户上传单张照片
- 单物体场景
- 俯拍
- 物体与标尺必须在同一平面
- 背景尽量干净，避免复杂干扰

首版不支持：

- 复杂背景鲁棒识别
- 多物体混拍
- 内部孔洞识别
- 上传 STL / OBJ 作为识别输入
- 从照片直接做 3D 重建

### 3.3 尺度校准方案

- 站内提供一个标准 `L` 形标尺 `STL` 供用户打印
- 用户拍照时，标尺与目标物体一起放置
- 系统基于该 `L` 形标尺完成像素与毫米换算

审查时必须展示：

- 标尺识别状态
- 标尺的已知尺寸说明
- 当前缩放比例

### 3.4 轮廓生成与审查

流程固定为：

1. 上传图片
2. 识别 `L` 形标尺
3. 提取目标物外轮廓
4. 生成简化后的关键点轮廓
5. 用户对关键点轮廓进行拖拽编辑
6. 基于审核后的轮廓进入几何生成

编辑边界：

- 只编辑“关键点轮廓”
- 不做像素级描边编辑
- 允许通过拖点修正轮廓

审查界面必须展示：

- 原始图片
- 标尺识别结果
- 轮廓叠加结果
- 整体宽度
- 整体高度
- 关键点尺寸标注或关键点毫米坐标

### 3.5 高度与盒体生成

- 照片只负责 `XY` 轮廓
- 物体高度由用户手工输入
- 系统基于审核后的轮廓和高度生成型腔
- 外部尺寸自动搜索只比较 `0°` 和 `90°` 两种摆放方向

目标：

- 先满足装得下
- 再尽量让外部体积最小
- 然后再兼顾占地和高度

### 3.6 取物凹槽

当前状态：`未实现`

首版必须支持 3 种模式：

- `双侧双层`
  - `X` 轴两侧都生成双层凹槽
- `单侧双层`
  - 由用户指定一侧生成双层凹槽
- `自动侧选择`
  - 系统根据轮廓和剩余空间自动决定使用单侧还是双侧

凹槽目标：

- 提高取物便利性
- 不明显破坏主体结构

### 3.7 架构方向

当前已落地：

- 独立的 `photo-outline-bin` 模板定义
- 专用的上传与轮廓审查界面，而不是直接复用现有通用参数表单
- 图像分析与几何生成职责分离
- 可序列化的复杂参数结构，用于承载轮廓点、标尺信息和审查结果

当前仍待补齐：

- 独立的图像识别 Worker

可行性参考：

- [Using OpenCV.js](https://docs.opencv.org/master/d0/d84/tutorial_js_usage.html)
- [Contours: Getting Started](https://docs.opencv.org/4.x/d5/daa/tutorial_js_contours_begin.html)
- [approxPolyDP](https://docs.opencv.org/4.x/d3/dc0/group__imgproc__shape.html)

## 4. 建议开发顺序

建议按下面顺序推进：

1. `参数列表界面 V2 与超宽屏适配`
   - 先把当前生成器主入口的参数组织和大屏体验收口，避免后续新增模板继续叠加旧表单结构
2. `旧收纳盒迁移`
   - 这是当前 STL 相关需求里唯一还没有真正起步的主要工作流
3. `照片轮廓收纳补完`
   - 重点补齐取物凹槽和独立图像识别 Worker
4. `钳类模板 V2`
   - 在 STL 与照片轮廓能力收口后，再回到工具类模板升级
