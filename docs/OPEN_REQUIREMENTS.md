# Open Requirements

更新时间：2026-03-11

这份文档用于沉淀“已讨论、已确认方向、但尚未完成”的开放需求，供后续开发、交接和 AI 上下文传递使用。

状态标记：

- `已实现`：当前代码已经落地，可在仓库中直接找到对应实现
- `部分实现`：已经有一部分能力，但仍未达到当前期望
- `开放需求`：已确认方向，但尚未完成开发
- `暂缓`：已讨论，但当前不进入近期排期

后续新增的未完成需求，默认继续追加到本文件，而不是分散到新的需求文档。

## 1. 当前代码基线

以下内容为当前真实已落地能力，用于和开放需求区分：

- 模板数量：4 个
  - `generic-bin`
  - `screwdriver-rack`
  - `memory-card-tray`
  - `pliers-holder`
- 生成器主路由：`/generator/:templateId`
- 参数系统：当前仍以标量参数为主，只支持 `string | number | boolean`
- 几何生成：浏览器内 Worker 执行
- 导出格式：仅支持 `STL`
- 3D 预览：`three.js`
- 预览坐标轴：已支持 `X / Y / Z` 坐标轴
- 预览原点：当前位于模型外轮廓角点，不再与模型中心点重合

当前模板和通用能力状态：

- `通用收纳盒`
  - `已实现`
  - 支持 `X / Y / Z` 三个方向内壁厚度调整
  - 支持按“从最外侧内壁到隔板起始边”的语义定义隔板距离
  - 支持统一设置隔板厚度和隔板高度
- `螺丝刀收纳`
  - `已实现`
  - 顶部孔位与内部腔体连通
- `内存卡托盘`
  - `已实现`
  - 当前稳定模式仅保留：
    - `microSD 极限收纳`
    - `SD 紧凑收纳`
    - `混合收纳`
  - 之前讨论过的竖放模式已取消，不视为当前需求基线
- `钳子收纳`
  - `部分实现`
  - 当前仍可用，但还没有升级为更通用的钳类模板 V2

开发和发布辅助能力：

- `已实现`：Windows 一键启动脚本 `scripts/start-dev.bat`
- `已实现`：IDE 可调用命令 `npm run dev:open`
- `已实现`：Linux 云服务器构建脚本 `scripts/build-release.sh`

## 2. 已确认但未完成的开放需求

以下需求都应视为 `开放需求`，尚未进入当前代码基线。

### 2.1 照片轮廓收纳模板

状态：`开放需求`

目标：

- 支持拍照上传
- 基于照片提取待收纳物品的外轮廓
- 自动生成最合适大小的 Gridfinity 盒子
- 可选在 `X` 轴两侧添加便于取物的双层凹槽
- 提供尺度校准与尺寸审查信息

备注：

- 这是当前最需要完整保留细节的开放需求
- 详细规格见第 3 节

### 2.2 型腔收纳盒

状态：`开放需求`

目标：

- 新增一个独立模板，先生成实体盒体
- 再在盒体内部挖出指定模型的收纳型腔
- 可选择被收纳物体的摆放方向
- 自动推荐最小外部尺寸，也允许锁定外部尺寸

当前边界：

- 不替换现有 `memory-card-tray`
- 预期先从站内参数化物品模板开始，而不是用户上传 3D 文件

### 2.3 钳类模板 V2

状态：`开放需求`

目标：

- 把当前“钳子收纳”升级为更通用的“钳类 / 手持夹持工具收纳”
- 用户只需要调整少数核心参数，就能适配更多钳类和类似工具
- 模板应从“固定矩形槽”升级为更通用的限位型或渐缩型槽道

建议的核心参数方向：

- 数量
- 最大宽度
- 最大厚度
- 插入长度
- 前部开口宽度

### 2.4 更复杂的非标输入能力

状态：`开放需求`

当前还没有落地的能力边界包括：

- 上传文件后参与几何求解的复杂参数结构
- 在模板里传递轮廓点数组或图像分析结果
- 独立的图像识别 Worker
- 轮廓拖点编辑型的专用模板界面

这些能力当前都不属于现有标量参数模板体系，需要新模板和新 UI 结构承接。

## 3. 照片轮廓收纳模板 V1 详细规格

状态：`开放需求`

### 3.1 模板定位

- 模板 id：`photo-outline-bin`
- 这是一个独立模板，与现有 4 个模板并列
- 不影响 `generic-bin`、`screwdriver-rack`、`memory-card-tray`、`pliers-holder`

### 3.2 输入与识别边界

首版固定为：

- 纯前端识别
- 用户上传单张照片
- 单物体场景
- 俯拍
- 物体与标尺必须在同一平面
- 背景尽量干净，避免复杂干扰

首版不支持：

- 复杂背景鲁棒识别
- 多物体混拍
- 内部孔洞识别
- 上传 STL / OBJ 作为识别输入
- 从照片直接做 3D 重建

### 3.3 尺度校准方案

- 站内提供一个标准 `L` 形标尺 `STL` 供用户打印
- 用户拍照时，标尺与目标物体一起放置
- 系统基于该 `L` 形标尺完成像素与毫米换算

审查时必须展示：

- 标尺识别状态
- 标尺的已知尺寸说明
- 当前缩放比例

### 3.4 轮廓生成与审查

流程固定为：

1. 上传图片
2. 识别 `L` 形标尺
3. 提取目标物外轮廓
4. 生成简化后的关键点轮廓
5. 用户对关键点轮廓进行拖拽编辑
6. 基于审核后的轮廓进入几何生成

编辑边界：

- 只编辑“关键点轮廓”
- 不做像素级描边编辑
- 允许通过拖点修正轮廓

审查界面必须展示：

- 原始图片
- 标尺识别结果
- 轮廓叠加结果
- 整体宽度
- 整体高度
- 关键点尺寸标注或关键点毫米坐标

### 3.5 高度与盒体生成

- 照片只负责 `XY` 轮廓
- 物体高度由用户手工输入
- 系统基于审核后的轮廓和高度生成型腔
- 外部尺寸自动搜索只比较 `0°` 和 `90°` 两种摆放方向

目标：

- 先满足装得下
- 再尽量让外部体积最小
- 然后再兼顾占地和高度

### 3.6 取物凹槽

首版必须支持 3 种模式：

- `双侧双层`
  - `X` 轴两侧都生成双层凹槽
- `单侧双层`
  - 由用户指定一侧生成双层凹槽
- `自动侧选择`
  - 系统根据轮廓和剩余空间自动决定使用单侧还是双侧

凹槽目标：

- 提高取物便利性
- 不明显破坏主体结构

### 3.7 架构方向

首版预期需要新增的能力：

- 独立的 `photo-outline-bin` 模板定义
- 专用的上传与轮廓审查界面，而不是直接复用现有通用参数表单
- 图像分析与几何生成职责分离
- 可序列化的复杂参数结构，用于承载轮廓点、标尺信息和审查结果

可行性参考：

- [Using OpenCV.js](https://docs.opencv.org/master/d0/d84/tutorial_js_usage.html)
- [Contours: Getting Started](https://docs.opencv.org/4.x/d5/daa/tutorial_js_contours_begin.html)
- [approxPolyDP](https://docs.opencv.org/4.x/d3/dc0/group__imgproc__shape.html)

## 4. 建议开发顺序

建议按下面顺序推进：

1. `照片轮廓收纳模板`
   - 这是当前最完整、最明确、也最需要独立架构承接的开放需求
2. `型腔收纳盒`
   - 可复用一部分“型腔生成”和“朝向控制”能力
3. `钳类模板 V2`
   - 在前两者稳定后再升级现有钳类模板
